# 项目方案：团队、环境、需求与过程

地标管理原型系统(Landmark Manager)面向校园、景区与城市探索场景,基于 Redis 7.2 的 GEO 能力提供地标录入、检索、地图展示与近邻查询。课程设计周期 1 周(5 工作日 + 周末缓冲),5 人小组以单 Sprint Scrum 节奏推进。

## 二、Redis 支撑环境搭建

目标版本 Redis 7.2(含 `GEOSEARCH` 命令)。开发宿主 Ubuntu 22.04 LTS 或 WSL2。

### 2.1 安装

源码编译,避免发行版仓库滞后:

```bash
sudo apt update && sudo apt install -y build-essential tcl pkg-config
wget https://download.redis.io/releases/redis-7.2.5.tar.gz
tar -xzf redis-7.2.5.tar.gz && cd redis-7.2.5
make -j"$(nproc)" && make test
sudo make PREFIX=/usr/local install
```

### 2.2 关键配置

`/etc/redis/redis.conf` 中按以下项调整(仅列改动行):

```conf
bind 127.0.0.1 ::1
port 6379
requirepass S3-redis-pass
appendonly yes
appendfsync everysec
save 300 10
maxmemory 512mb
maxmemory-policy allkeys-lru
```

| 配置项                                | 含义       | 取值原因                              |
| ------------------------------------- | ---------- | ------------------------------------- |
| `bind`                                | 监听地址   | 课设阶段限本机,避免暴露              |
| `requirepass`                         | 客户端密码 | 启用 AUTH,配合应用统一管理          |
| `appendonly` + `appendfsync everysec` | AOF 持久化 | 兼顾性能与最多 1 s 数据丢失           |
| `save 300 10`                         | RDB 触发   | 5 分钟内 10 次写则快照,作为兜底     |
| `maxmemory-policy allkeys-lru`        | 淘汰策略   | 适合缓存与 GEO 共存场景               |

systemd 托管:

```bash
sudo cp utils/systemd-redis_server.service /etc/systemd/system/redis.service
sudo systemctl daemon-reload && sudo systemctl enable --now redis
```

### 2.3 GEO 命令冒烟测试

```bash
redis-cli -a S3-redis-pass <<'EOF'
GEOADD landmarks:beijing 116.397 39.908 "天安门"
GEOADD landmarks:beijing 116.310 39.992 "颐和园"
GEODIST  landmarks:beijing "天安门" "颐和园" km
GEOSEARCH landmarks:beijing FROMMEMBER "天安门" BYRADIUS 20 km ASC
EOF
```

预期:`GEODIST` 返回约 14 km;`GEOSEARCH` 按距离升序列出半径内地标。

| 层         | 选型                                                  | 版本             |

| 后端框架   | FastAPI + redis-py(asyncio)+ Pydantic v2          | 0.115 / 5.x / 2.x |

## 四、非功能需求

### 4.1 性能

| 指标         | 目标                                | 验证方式                |
| ------------ | ----------------------------------- | ----------------------- |
| 近邻查询响应 | P95 < 100 ms(≤1 万地标,半径 5 km) | wrk 100 并发,80 % 读  |
| 写入响应     | P95 < 50 ms                         | wrk 写入剧本            |
| 单实例吞吐   | ≥ 1500 QPS                          | wrk 60 s 压测           |
| 首屏渲染     | < 2 s(4G 节流)                    | Chrome DevTools         |

### 4.2 可靠性

- 演示口径下连续 8 h 无重启,服务可用性 ≥ 99 %
- Redis 同时启用 AOF (`appendfsync everysec`) 与 RDB(5 分钟兜底快照),最多丢失 1 s 数据
- 关键操作落地结构化日志(JSON Lines),保留 ≥ 7 天

### 4.3 安全

- 密码使用 BCrypt(cost ≥ 10)哈希,禁止明文与 MD5
- 鉴权采用 JWT(HS256,带 `jti` 声明),24 h 过期;登出后 `jti` 写入 Redis 吊销集
- 登录限流:5 次 / 分钟 / IP,超出返回 429
- 全部 Redis 命令通过 redis-py 参数化执行,杜绝命令注入;前端输出统一转义,防 XSS
- Redis 仅监听内网回环并启用 `requirepass`;CORS 白名单仅放行前端站点

### 4.4 可用性与可维护性

- 兼容 Chrome / Firefox / Edge 100+,中文界面,错误提示明确不暴露堆栈
- 后端按 `routes / repository / schemas / security` 分包,接口契约由 OpenAPI 自动生成,Swagger UI 实时可访问
- 单元测试行覆盖 ≥ 60 %,GEO 服务关键路径 ≥ 80 %;CI 中执行 `ruff` + `pytest`

### 4.5 可扩展性

- 地标按城市分 Key (`landmarks:city:{cityId}`),便于后续切到 Redis Cluster 时按 hashtag 路由
- 配置外置(`.env` + Pydantic Settings),开发与演示双套切换
- 前端瓦片层抽象,可在 OSM / MapBox / 高德间切换

## 五、过程管理:Scrum

### 5.1 选用理由

- 一周交付原型,需求随评审滚动调整,瀑布式难以容纳变更
- 5 人小组落在 Scrum 推荐的 3–9 人区间,沟通成本可控
- 事件节拍清晰,易于在课程节点(开题 / 中期 / 答辩)对齐

角色分工见 §1.1。PO、SM 均不脱产,体现小团队下的"自组织"原则。

### 5.2 一周事件节拍

| 事件                 | 时长   | 时间       | 交付物                          |
| -------------------- | ------ | ---------- | ------------------------------- |
| Sprint Planning      | 90 min | D1 09:00   | Sprint Backlog、容量评估        |
| Daily Standup        | 10 min | 每日 09:30 | 进展、阻塞、当日承诺            |
| Mid-Sprint Sync      | 30 min | D3 17:00   | 风险评估、范围调整              |
| Sprint Review        | 40 min | D5 16:00   | 可演示增量、PO 验收单           |
| Sprint Retrospective | 20 min | D5 16:40   | 改进项与 owner                  |

### 5.3 工件与看板

- **Product Backlog**:用户故事按业务价值排序,PO 维护
- **Sprint Backlog**:本周承接的故事 + 任务卡(粒度 ≤ 8 h),GitHub Projects 看板呈现,随站会更新
- **增量**:每日合入 `main` 的可运行版本,本地一键启动 (`uvicorn` + `vite`) 演示
- **燃尽**:按剩余任务卡数手工记录,Mid-Sprint 偏离 > 20 % 触发范围调整

### 5.4 完成定义

| 阶段               | 准则                                                                          |
| ------------------ | ----------------------------------------------------------------------------- |
| 进入 Sprint(DoR) | 满足 INVEST、验收准则明确、估点 ≤ 5、外部依赖可解                            |
| 故事完成(DoD)    | 代码合入 `main`、`pytest` 通过、Swagger 同步、本地冒烟通过、PO 验收签字       |

### 5.5 风险与对策

| 风险                                  | 影响 | 应对                                |
| ------------------------------------- | ---- | ----------------------------------- |
| Redis 版本差异(`GEOSEARCH` 自 6.2) | 高   | 锁定 7.2,启动脚本断言版本号       |
| 前后端联调延期                        | 高   | D2 起前端跑 Mock 数据,D3 切真实接口 |
| 地图瓦片配额不足                      | 中   | 主用 OSM,备用本地 tileserver       |
| 需求蔓延                              | 中   | D5 09:00 起冻结新功能,仅修 Bug      |
| 成员临时缺席                          | 中   | 任务卡双 owner,关键模块结对编程     |
